### **Chapter 3：威脅建模 — 為什麼 AI 寫的 Code 會讓你賠錢？**
**核心目標**：利用「恐懼」來強化「規範」的重要性。這章不寫 Code，而是當駭客（Red Team）。

#### **3-1 中間人攻擊 (MITM)：篡改風險**
*   **整合策略**：採用 Version B 的「攻防對照」結構。
*   **攻擊 (Attack)**：演示如何攔截瀏覽器封包，將 `amount: 1000` 改為 `amount: 1`。
*   **防禦 (Defense)**：引入 **檢查碼 (CheckMacValue / Signature)** 的概念。
    *   **比喻**：就像古代的「虎符」或現代的「騎縫章」，證明信件途中沒被拆過。

#### **3-2 偽造通知 (Fake Webhook)：身分風險**
*   **整合策略**：解決「門戶大開」的問題。
*   **攻擊 (Attack)**：駭客用 Postman 模擬金流商格式，直接對你的 API 發送「付款成功」。你的 Server 傻傻地就發貨了。
*   **防禦 (Defense)**：
    1.  **來源檢查**：只相信來自金流商 IP 的請求。
    2.  **驗簽 (Verify Signature)**：確保這封信真的是金流商簽名的。

#### **3-3 髒資料地獄 (Dirty Data)：一致性風險**
*   **整合策略**：解決 Version A 提到的「掉單」與「重複扣款」。
*   **情境**：
    *   **掉單**：使用者付錢了，但 Webhook 沒送達（網路波動）。
    *   **重複**：金流商以為你沒收到，同樣的 Webhook 送了 5 次。
*   **防禦 (Defense)**：
    *   **查單機制 (Re-query)**：主動去問金流商狀態。
    *   **冪等性 (Idempotency)**：同樣的通知收 100 次，資料庫也只會記 1 筆成功，不會發 100 次貨。

    *   **觀念：** 我們不用自己設計複雜的 Table，但要告訴 AI 我們「必須記住」哪些資訊。

    *   **必備資訊 2 - 交易狀態 (Status)：**
        *   *邏輯：* 因為交易有時間差（已建立、付款中、成功、失敗），我們需要一個欄位來記錄「現在走到哪一步」，才能決定要不要給商品。
    *   **必備資訊 3 - 交易驗證碼 (TradeInfo/MacValue)：**
        *   *邏輯：* 因為 Ch 3 說會有駭客竄改，所以我們必須把加密後的字串存下來（或傳出去），用來比對資料是不是原裝的。
*   **4-3 產出規格書 (The Spec)**
    *   **實作：** 總結上述重點，列出一份「Prompt 關鍵字清單」。

#### **3-3 髒資料地獄 (Dirty Data)：一致性風險**
*   **整合策略**：解決 Version A 提到的「掉單」與「重複扣款」。
*   **情境**：
    *   **掉單**：使用者付錢了，但 Webhook 沒送達（網路波動）。
    *   **重複**：金流商以為你沒收到，同樣的 Webhook 送了 5 次。
*   **防禦 (Defense)**：
    *   **查單機制 (Re-query)**：主動去問金流商狀態。
    *   **冪等性 (Idempotency)**：同樣的通知收 100 次，資料庫也只會記 1 筆成功，不會發 100 次貨。
