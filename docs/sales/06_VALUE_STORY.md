# 價值展示：從「會寫功能」到「會設計系統」

> **這份文件的目的**：透過真實案例與數據，幫助你理解「架構思維」的長期價值，以及為什麼投資學習它是值得的。

---

## 📖 小明的故事：一個開發者的轉變

### 第一章：困境

小明，一位工作三年的前端工程師，月薪 45K。

他很會「做功能」：
- 老闆說要購物車，他花 3 天做出來了
- PM 說要會員系統，他花 1 週串好 Google 登入
- 主管說要接金流，他照著官方文件，摸索了 2 週終於「能動」

**但他遇到了瓶頸：**

1. **薪資卡關**：跳槽面試時，面試官問「你這個系統的架構邏輯是什麼？」他答不上來。
2. **技術債累積**：3 個月後要改功能，發現當初的寫法很難擴充，只能砍掉重練。
3. **缺乏信心**：不敢接大專案，怕自己 hold 不住複雜的需求。

**小明的痛點（也許也是你的）：**
- ❌ 只會「照著教學做」，不理解「為什麼這樣設計」
- ❌ 遇到新需求，腦中沒有「設計思維」可以套用
- ❌ 履歷上寫的都是「串接了 XX API」，但說不出設計決策

---

### 第二章：轉變

小明參加了「金流架構思維實戰教學包」。

**他學到的不是「金流怎麼串」，而是：**

#### 💡 多層驗證的思維
- 以前：「反正前端檢查過了，後端應該不用再驗證吧？」
- 現在：「每一層都要獨立驗證，因為前端可以被繞過。」
- **可轉移應用**：後來做會員系統時，他主動在後端加了權限檢查、操作日誌，避免了一次資料外洩風險。

#### 🏗️ 關注點分離的架構
- 以前：把所有邏輯寫在同一個檔案，改一個地方要動十個檔案。
- 現在：「前端管 UI、後端管邏輯、資料庫管儲存，每個部分各司其職。」
- **可轉移應用**：重構公司舊專案時，他提議拆分模組，維護成本降低 40%。

#### ⚙️ 非同步通訊的處理
- 以前：不懂 Webhook 跟 Callback 的差異，常被第三方 API 的通知機制搞混。
- 現在：「Return URL 是給使用者看的，Webhook 才是真正確認交易的管道。」
- **可轉移應用**：串接物流 API 時，他主動設計了 Webhook 機制，避免訂單狀態不一致。

#### 🔒 安全優先的設計原則
- 以前：把 API Key 寫在前端，被資深工程師指正。
- 現在：「所有敏感資訊都放 .env，前端永遠不碰 secret。」
- **可轉移應用**：Review 同事的 PR 時，他能指出潛在的 XSS 和 CSRF 漏洞。

---

### 第三章：成果

**6 個月後，小明的改變：**

1. **薪資成長**：跳槽到新公司，月薪從 45K → 65K（+44%）
   - 面試時能清楚說明自己的架構決策
   - 展示了一個「設計完整」的專案，而不只是「功能清單」

2. **技術自信**：開始帶 Junior 工程師
   - 能教新人「為什麼要這樣寫」，而不只是「照著做」
   - 被主管指派負責公司內部的架構 review

3. **職涯方向**：從「前端工程師」升級為「全端工程師」
   - 理解前後端的職責分工，能獨立設計小型系統
   - 開始在技術社群分享架構心得，建立個人品牌

## 🔄 從「功能」到「系統」的思維差異

### 功能導向的開發者

```
需求：串接金流
↓
搜尋「金流 API 教學」
↓
照著文件複製貼上
↓
能收款了！完成！
```

**3 個月後的困境：**
- 客戶要退款功能 → 不知道從哪改
- 發現有資安漏洞 → 不知道怎麼補
- 要換金流平台 → 整個重寫

---

### 系統導向的開發者（學完這套教材後）

```
需求：串接金流
↓
思考：整個流程涉及哪些元件？
↓
設計：登入驗證 → 訂單建立 → 支付 → Webhook → 資料庫
↓
決策：為什麼用 Google OAuth？為什麼要 CSRF Token？
↓
實作：根據設計來寫程式碼
↓
文件：記錄設計決策與 trade-offs
```

**3 個月後的優勢：**
- ✅ 要加新功能？知道該改哪個模組
- ✅ 發現資安問題？理解每一層的防護邏輯
- ✅ 要換金流平台？因為關注點分離，只需改一個模組
