# Docs Navigator

> **這份總覽的任務**：讓不同角色可以在 1 分鐘內找到自己該看的文件，知道閱讀順序、時間成本與輸出成果。

---

## 🎯 使用方法

1. **先確認你當下的目標**（入門／開發／上線／營運），對照下方旅程表挑選必讀章節。
2. **閱讀每份文件前**，先看這份總覽給的 `TL;DR`、預估時間與輸出成果，避免一次啃完全部。
3. **若你是內部營運或銷售**，請直接跳到「Internal / Sales」專區。

> 建議配合 VS Code 的分頁，主視窗打開本總覽，右側預覽實際文件，快速往返。

## 🔄 設計與維護哲學

### 教學邏輯
本教材採用「**具體問題 → 通用原則 → 實踐應用**」的三段式設計：
1. **Stage 00 (為什麼要學)**：提出市場價值 & 個人動機
2. **Stage 05-1.5 (前置+理念)**：用「4 大挑戰」引出「4 個設計原則」
3. **Stage 2-4 (技術+實踐+部署)**：從理論到程式碼，每步都回溯「為什麼這樣設計」

### 資訊密度曲線
- **Stage 1-1.5**：低密度，建立基礎
- **Stage 2**：中密度，技術細節（分成 01 理論 + 02 實踐）
- **Stage 3-4**：中高密度，實作與除錯
- **Stage 5-6**：高密度，進階優化與挑戰

### 新增文件前的檢查清單
1. **這份文件解決什麼「具體問題」**？（不只是列舉技術）
2. **應該放在哪個 Stage**？（是否符合資訊密度曲線）
3. **與前後文的銜接點在哪**？（每份文件應有「為什麼要讀它」的說明）
4. **預估時間合理嗎**？（避免單篇超過 90 分鐘）

本導航已依據 `AI_OUTPUTS/Pivoted_Payment_Gateway_Strategy.md` 重新設計，核心從「銷售功能」轉向「傳遞思維」。所以：

- 每個防線都說「它解決什麼威脅」而非純技術細節

若您發現：
- ❌ 某文件讀後還是茫然 → 可能「為什麼」說明不足

- ❌ 某些概念前後矛盾 → 可能銜接點有問題
